CodeCacheHost

mojolpm 在每次调用 RunTestcase时，都会先运行 setup，来设置该 testcase 的运行环境。

关键点是SetUpOnUIThread这个函数，其实很明显的这里所设置的环境都是为了后续的操作而服务的。

在 code_cache_host 例子中，首先生成了browser_context_，借助该环境来获取一系列参数完成cache_storage_control_wrapper_的实例化，最后在初始化generated_code_cache_context_。

而且这些数据都最终被用于AddCodeCacheHostImpl，该函数则处于

其实testcase 的所有初始化操作都最终服务与Action::kNewCodeCacheHost，来生成 CodeCacheHost 的实例。与之对应的我们可以看其源码，总结出 testcase 的初始化过程服务于被测用例的各个接口的需求。

除了这些初始化操作不同外，比较特别的还包含于 testcase 的初始化

关于这一点，在官方文档上有所提及：At this point, we realise that it’s likely that we want to be able to have multiple CodeCacheHostImpl‘s with different render_process_ids and different backing origins。

FileSystemManager

与之前 CodeCacheHost 不同的地方在于，FileSystemManager 既执行了 SetUpOnIOThread，又执行了 SetUpOnUIThread。其整体行为还是初始化先决条件，在其源文件可以发现，该类的所有方法都必须在 IO 线程上调用(构造函数除外)，析构函数同样也要在 IO 线程上运行，所以要分别设置好IO 与 UI 的环境。不一样的是，FileSystemManager，把实例化FileSystemManagerImpl的过程移植到了 SetUpOnUIThread 函数内，侧面印证了构造函数要在 UI 线程运行，其余函数都在 IO 线程运行。

AddFileSystemManager & AddFileSystemManagerImpl 完成生创建 remote，将 remote 绑定到一个新的 pipe，将 receiver 端绑定到file_system_manager_impls_[offset]，file_system_manager_impls_内部则存储着FileSystemManagerImpl的实例，以此来接收remote 发送的信息。

这些定义了 fuzzer 能够执行的行为，除了原本FileSystemManager所包含的行为外，还包含了FileSystemCancellableOperation，根据文档描述，我们的 fuzzer 能够支持更多接口，如下：

Additional fuzzable interfaces

  • 该接口返回一个 PendingRemote<>

  • 该接口接受一个 PendingReceiver<>

Renderer-hosted interfaces

如果一个接口方法需要一个 PendingRemote<>(或者返回一个 PendingReceiver<>) ，那么我们还需要向 fuzzer 添加响应处理。This lets the fuzzer send fuzzer-side implementations of mojo interfaces, and handle fuzzing the values returned if those methods are called.

小结

  • mojolpm 除了完成默认的环境配置外，还应该根据要 fuzz 的接口的需求和特定要求，在每个 testcase run 之前完成特定的初始化任务

  • 均应包含 默认类的实例化工作，add 函数主要完成 receiver 端的绑定工作

  • 整体逻辑大致如下：

    • 实例化测试类，作为browser端，创建消息管道，与receiver端绑定
    • 运行线程（环境）
    • 将测试用例与 remote 端绑定，发送数据
