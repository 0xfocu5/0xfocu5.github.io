Protocol Buffers

Protocol Buffers 是一种与语言无关，平台无关的可扩展机制，用于序列化结构化数据。使用Protocol Buffers 可以一次定义结构化的数据，然后可以使用特殊生成的源代码轻松地在各种数据流中使用各种语言编写和读取结构化数据。Protocol Buffers 目前有2和3两个版本号。

环境搭建

Demo

通过 protoc -I=. --cpp_out=. demo.proto 来生成对应文件。指定 cpp后，会生成.pb.cc .pb.h两个文件。

-I：指定protoc的搜索import的proto的文件夹

cpp_out：用来生成C++代码

更多的命令可以参考，protoc –help

在上述 deom 中，我们定义了一个Person的消息类型，通过syntax来指定 proto的版本为proto2，若不指定则默认为 proto2。person消息定义三个字段(name、ID、email)，每个字段都指定了名称和类型。这些都是基本的标量类型，当然我们也可以定义枚举类型、复合类型和其他消息类型等。

修饰符

  • optional: 该字段可能会或可能不会被设置。如果未设置可选字段值，则使用默认值。对于简单类型，可以指定自己的默认值，就像我们type在示例中为电话号码所做的那样。否则，使用系统默认值：数字类型为零，字符串为空字符串，布尔值为 false。对于嵌入的消息，默认值始终是消息的“默认实例”或“原型”，它没有设置任何字段。调用访问器以获取尚未显式设置的可选（或必需）字段的值始终返回该字段的默认值。
  • repeated：该字段可以重复任意次数（包括零次）。重复值的顺序将保留在协议缓冲区中。将重复字段视为动态大小的数组。
  • required：必须提供该字段的值，否则消息将被视为“未初始化”。如果libprotobuf在调试模式下编译，序列化未初始化的消息将导致断言失败。在优化的构建中，会跳过检查，无论如何都会写入消息。但是，解析未初始化的消息总是会失败（通过false从 parse 方法返回）。除此之外，必填字段的行为与可选字段完全相同。

fieldNumber

optional string name = 1; 这里的1(fieldNumber),不是字段name(fieldName)的值, 而是字段name的Tag.(即: 在编码后1 就代表了 name这个字段).

NOTE: fieldNumber一旦被使用， 终生这个编号都不要改变。fieldNumber 的取值范围是1~2^29-1。而常用的fieldNumber范围是: 1-15(只用1个byte编码), 16-2047(采用2个byte编码).

字段定义规则

消息字段可以是以下之一：

singular单数字段: protobuf的默认字段规则, 就是说这个字段只能出现0或者1次.

repeated重复字段: 就是该字段是一个数组或者list. 数组里面可以有任意数量的元素. 如果有多个元素, 元素的顺序会被保留.

在proto3中，repeated标量数字类型的字段为默认使用编码。

在proto2中，

  1. required：必须有一个
  2. optional：0或者1个
  3. repeated：任意数量（包括0）

枚举类型

枚举定义在一个消息内部或消息外部都是可以的，如果枚举是 定义在 message 内部，而其他 message 又想使用，那么可以通过 MessageType.EnumType 的方式引用。

定义枚举的时候，我们要保证第一个枚举值必须是0，枚举值不能重复，除非使用 option allow_alias = true 选项来开启别名。

package

定义proto的包名，包名可以避免对message 类型之间的名字冲突，同名的Message可以通过package进行区分。在没有为特定语言定义option xxx_package的时候，它还可以用来生成特定语言的包名，比如Java package, go package。

import

引入其他 proto 文件。比较少使用的是public和weak关键字。默认情况下weak引入的文件允许不存在(missing)，只为了google内部使用。public具有传递性，如果你在文件中通过public引入第三方的proto文件，那么引入你这个文件同时也会引入第三方的proto。

oneof

如果您有一个包含许多字段的消息，并且最多只能同时设置一个字段，则可以使用oneof功能强制执行此行为并节省内存。
